# Test Design: Harden Security Applicability Contract

## Overview

This change hardens SDD security applicability governance rather than application runtime behavior. Test evidence is therefore static and manual: the repository has no configured runtime test runner, linter, type checker, formatter, or coverage command, and no runtime command should be invented. Once implemented, the planned PowerShell validator `scripts/validate_security_applicability.ps1` provides static pass/fail evidence for `security-applicability.md`; manual contract inspections cover skill, shared-contract, catalog, downstream compatibility, and archive-evidence obligations.

## No-Impact Assessment

This change is not no-impact: `security-applicability.md` classifies it as `security-impacting` with `securityImpact: true`, and `security-design.md` records mandatory controls for all supported taxonomy categories. No-impact behavior is still part of the test surface because the specs require enriched applicability fields to preserve valid no-impact routing for future changes. TD-004, TD-005, TD-019, and TD-020 cover that compatibility requirement.

## Evidence Expectations

- Mandatory cases are verification-blocking when uncovered, unexecuted, or not replaced by a complete approved exception.
- Static validator evidence must be collected only after `scripts/validate_security_applicability.ps1` is implemented; until then, tasks should create the script before verify expects command output.
- Manual evidence must identify exact inspected files and the contract fields verified; broad statements such as "reviewed docs" are insufficient.
- Runtime test, coverage, linter, type-checker, and formatter commands are unavailable in `openspec/config.yaml`; verification must report this explicitly and must not claim missing tooling as pass evidence.
- Validator evidence must stay scoped to Markdown/YAML structure, routing consistency, matrix completeness, guideline/source references, overrides, severity vocabulary, no-impact proof, and validation metadata. It must not claim runtime authentication, session, data, secret, access-control, file, database, or logging security behavior.
- Archive readiness requires preserving the security-design archive gate fields: catalog snapshot identity, guideline IDs, taxonomy categories, source references, operational severity, expected evidence status, residual risks, and exception state.
