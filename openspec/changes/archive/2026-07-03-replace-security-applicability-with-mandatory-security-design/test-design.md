# Test Design: Mandatory Security Design for SDD Changes

## Overview

This change redesigns SDD security governance rather than application runtime behavior. The test plan therefore uses static and manual evidence only: the repository has no runtime test runner, package manifest, linter, formatter, type checker, or coverage command. Planned checks focus on proving the new DAG, state/persistence tokens, OpenSpec conventions, skill/agent contracts, validator migration, README/spec synchronization, legacy compatibility, and mandatory security controls from `security-design.md`.

## No-Impact Assessment

This change is not a no-impact change. It is governance/security-impacting because it changes mandatory SDD security planning, review, persistence, and archive gates. Runtime application no-impact categories remain N/A: authentication, sessions, file handling, and database access are not modified; runtime tests, runtime logging checks, database tests, auth/session tests, and file-upload/download tests are therefore not applicable. Those N/A categories still require preserved rationale and evidence through TD-027 rather than silent omission.

## Evidence Expectations

- Mandatory test cases are verification-blocking when uncovered. Non-mandatory cases, if added later, should be reported as warnings when uncovered.
- Evidence is static/manual by design. The repository has no configured runtime test runner, linter, formatter, type checker, coverage command, unit layer, integration layer, or E2E layer.
- Verification MUST cite unavailable runtime tooling explicitly and MUST NOT treat missing commands as passing evidence.
- Static evidence should include read-back excerpts, grep summaries, file existence checks, and future PowerShell validator summaries where scripts exist after apply.
- `scripts/validate_security_design.ps1` behavior should be validated with positive and negative fixture artifacts once implemented; until then, manual schema/vocabulary checks are acceptable planned evidence, not completed validation.
- Stale active-routing wording checks must distinguish allowed legacy/archive references from prohibited new-change routing authority.
- `review-security-report.md` must exist, be readable, and state a non-blocking or blocking verdict before verify/archive. Verify/archive evidence must cite it without duplicating its matrix.
- Security evidence must use paths, section names, summaries, and redacted placeholders. It must not print raw secrets, credentials, PAN, PII, or unnecessary confidential context.
- N/A security rows must preserve rationale and evidence location. Absence of runtime behavior is not enough unless documented in `security-design.md`, review-security evidence, or verify/archive evidence.
