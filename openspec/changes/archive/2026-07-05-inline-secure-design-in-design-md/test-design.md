# Test Design: Inline Secure Development Design in `design.md`

## Overview

This change modifies SDD workflow governance artifacts, not application runtime behavior. Test evidence is therefore static/manual: inspect Markdown contracts, OpenSpec specs, state routing, and future review evidence. No runtime test runner, coverage tool, linter, type checker, or formatter is configured in `openspec/config.yaml`, so unavailable tooling is a constraint and not passing evidence.

## No-Impact Assessment

Not applicable. The change has no application runtime behavior impact, but it has substantial SDD workflow, routing, state/persistence, security governance, review, and archive compatibility impact. Static/manual evidence is required for those contracts.

## Evidence Expectations

- Mandatory cases require implementation, execution where tooling exists, static/manual evidence, or a justified skip. In this repository, planned evidence is static/manual because runtime and quality commands are unavailable.
- Non-mandatory cases should be reported as warnings when uncovered, but they do not block verification by themselves.
- Security-design validation evidence must cite embedded `design.md` metadata: catalog snapshot, catalog/taxonomy version, guideline ID, applicability, lifecycle status, evidence owner, expected evidence, residual risk, and exception data when present.
- For this change, absence of standalone `security-design.md` is not a blocker. The authoritative security input is `design.md#secure-development-design`; standalone `security-design.md` is legacy/archive compatibility data only.
- Active new-change validation must not require `scripts/validate_security_design.ps1`. Any retained reference must be legacy/archive-only and non-blocking for status, review-security, verify, and archive.
- Runtime tests, linters, type checkers, formatters, and coverage commands that are unavailable must be reported as unavailable evidence, not treated as passed checks.
- Verification should read back changed artifacts and state before declaring success, especially route tokens, artifact refs, and security review gates.
