# Test Design: Selective Secure Design Controls

## Overview

This change modifies SDD instruction contracts, shared security wording, adapter prompts, and OpenSpec source specs so `design.md#secure-development-design` becomes a narrative changed-surface and applicable-category rules section. It deliberately removes design-time YAML, schema, control-table, compact-matrix, Source ID matrix, and all-row `N/A` requirements while keeping `review-security-report.md` as the exhaustive machine-readable validation owner. The repository has no configured runtime test runner, build command, linter, type checker, formatter, or coverage command, so all planned validation is static/manual read-back evidence; unavailable tooling is reported as unavailable evidence, never as passing evidence.

## Source ID Coverage Baseline

Corporate source-row coverage applies because the change modifies SDD security workflow contracts and security evidence gates. Test design consumes catalog context by reference only: `skills/_shared/security-guideline-catalog.md`, expected corporate inventory `expectedSourceIdCount: 155`, compact category taxonomy, lifecycle/status vocabulary, safe-evidence policy, and owner-phase boundary. It does not require `design.md#secure-development-design` to carry compact mappings, Source ID rows, Source ID groups, machine-readable status fields, or exhaustive `N/A` decisions.

The exhaustive source-row baseline is a planned `review-security-report.md` obligation:

- `review-security-report.md` must materialize every expected Source ID exactly once from the catalog inventory.
- It must own `Yes`/`No`/`N/A` decisions, compact mappings, lifecycle/status fields, evidence observations, findings, blockers, and exceptions.
- It must validate missed applicable categories by correlating proposal/spec/design/test-design/apply/review evidence, not by trusting omitted design rows.
- It must justify every report-level `N/A` row with safe irrelevance evidence and block unsupported `N/A`, missing/duplicate/unknown Source IDs, missing compact mappings, unsafe evidence, and incomplete exceptions.
- Warning/advisory findings must remain visible to verify/archive when mandatory evidence is complete.

## No-Impact Assessment

Not applicable. The change has no runtime application behavior impact, but it is security-impacting because it changes SDD security planning, testing, review, verification, archive, and evidence ownership contracts. Static/manual checks are mandatory for the changed Markdown contracts, adapter prompts, and source specs.

## Evidence Expectations

- Mandatory cases require static/manual read-back evidence, implementation evidence, or a complete approved exception before verify/archive can pass.
- Non-mandatory cases should be reported as warnings when uncovered, but they do not block verification by themselves.
- Test-design consumes only the narrative changed-surface classification and applicable category rules from `design.md#secure-development-design`.
- `design.md#secure-development-design` must not be validated as a YAML/schema/control-table/compact-matrix/Source ID matrix artifact for new active changes.
- Review-security owns the machine-readable report schema, exhaustive compact-control matrix, exhaustive Source ID matrix, exact-once `expectedSourceIdCount: 155` validation, report-level `N/A` decisions, missed applicable category/control blockers, unsafe-evidence blockers, exceptions, verdict, and next recommendation.
- Catalog context is used by reference for taxonomy, path/snapshot identity, inventory count, lifecycle/status vocabulary, safe-evidence policy, and review-security expansion; this artifact must not duplicate the full 155-row matrix.
- Unsupported `N/A`, missing/duplicate/unknown Source IDs, missing compact mappings, missing mandatory evidence, unsafe evidence, malformed report schema, and incomplete exceptions must block in `review-security-report.md`.
- Warning-only or advisory evidence must remain visible to review-security, verify, and archive when mandatory evidence is complete.
- Historical exhaustive design/security artifacts remain compatibility data and must not be rewritten, invalidated, normalized into active successors, or treated as required output for new changes.
- Runtime tests, build commands, linters, type checkers, formatters, and coverage commands are unavailable in this repository and must be reported as unavailable evidence, not passed checks.
- Evidence must cite safe paths, sections, summaries, command summaries, or redacted placeholders only; it must never include raw secrets, PII, PAN, tokens, credentials, connection strings, private keys, or unnecessary confidential values.
