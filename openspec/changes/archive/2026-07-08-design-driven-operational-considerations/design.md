# Design: Design-Driven Operational Considerations

## Technical Approach

Replace the contract-driven operational-readiness model with a design-owned operational-considerations model. The implementation deletes `skills/_shared/sdd-operational-readiness-contract.md`, removes active references to it, and updates SDD phase skills plus active OpenSpec specs so later phases consume only operational evidence that actually appears in `design.md`, `test-design.md`, `tasks.md`, apply evidence, review reports, verify reports, archive reports, or archived artifacts.

The change is documentation/instruction-contract work in an AI agent/skill repository. There is no application runtime, database, API, or executable test runner in `openspec/config.yaml`; validation will therefore rely on static inspection and repository search evidence.

## Architecture Decisions

### Decision: Delete the readiness contract instead of preserving compatibility

**Choice**: Delete `skills/_shared/sdd-operational-readiness-contract.md` and remove active references.
**Alternatives considered**: Keep it as optional guidance or archive compatibility.
**Rationale**: The proposal explicitly makes deletion in scope and preserving the file out of scope. Keeping the file would leave an attractive stale authority and make future phase behavior ambiguous.

### Decision: Make operational applicability design-owned

**Choice**: `sdd-design` may include `## Operational Considerations` only when the change needs operational behavior planning; downstream phases derive checks, tasks, review, verify, and archive behavior from actual artifact evidence.
**Alternatives considered**: Keep mandatory `## Operational Readiness` with all categories evaluated.
**Rationale**: Mandatory category completeness inflated every change with operational bookkeeping. Design-owned applicability preserves useful operational planning without forcing absent categories.

### Decision: Preserve safe-evidence controls outside the deleted contract

**Choice**: Move restricted operational data guidance into existing security/safe-evidence contracts and phase-local wording where evidence exists.
**Alternatives considered**: Remove operational safe-evidence language together with readiness gates.
**Rationale**: The old completeness gate is being removed, not the protection against leaking production identifiers, secrets, payloads, raw logs, full ID lists, or generated bytes.

### Decision: Keep `sdd-operational-doc` manual and archive-consuming

**Choice**: Update `sdd-operational-doc` to generate from archived evidence when present, mark inapplicable content as `No aplica.`, and mark missing applicable content as `Pendiente de confirmar:` without blocking on missing readiness evidence.
**Alternatives considered**: Make operational-doc a DAG phase or require archived readiness evidence.
**Rationale**: Project instructions define `sdd-operational-doc` as a manual post-archive utility, not a required SDD phase.

## Data Flow

After the change, operational evidence flows from authored artifacts, not from a global readiness contract:

```text
proposal/specs
   └─> design.md
         ├─ optional ## Operational Considerations
         └─ ## Secure Development Design
              └─> test-design.md ──> tasks.md ──> apply evidence
                                      └─> review-report.md
                                           └─> review-security-report.md
                                                └─> verify-report.md ──> archive
                                                                      └─> manual sdd-operational-doc
```

If `design.md` omits operational considerations or states `No aplica.` with rationale, later phases must not synthesize mandatory readiness categories. If any artifact does include operational evidence or gaps, later phases must preserve and validate that evidence according to its owner phase and safe-evidence boundaries.

## Interfaces / Contracts

No runtime API or data model changes are required. The changed contracts are Markdown skill/spec instructions:

- Active phases must not read, cite, or validate against `skills/_shared/sdd-operational-readiness-contract.md`.
- `## Operational Considerations` is conditional and design-owned. It may describe applicable logs/errors, monitoring mechanisms, administration operations, reprocessing/recovery, backup/retention/cleanup/generated artifacts, final-document boundaries, evidence owners, and unresolved gaps.
- Downstream phases must consume actual artifact evidence only. They may review missing applicable evidence when the design or implementation makes it applicable, but they must not require all legacy readiness categories.
- Exact `Pendiente de confirmar:` and `No aplica.` remain safe marker states for applicable missing or inapplicable operational content.

## Operational Considerations

This change has operational considerations for the SDD workflow itself, not for an application runtime. The implementation must prevent stale process gates from reappearing: all active skill/spec references to the deleted contract and mandatory readiness completeness language should be removed or rewritten. Monitoring-related wording must remain mechanism-oriented when operational evidence is applicable and must not default to SQL-only checks.

The manual `sdd-operational-doc` utility must continue to preserve sections 1-9 and diagrams R1-R4, but it must generate from archive evidence rather than a required readiness section. If archived operational evidence is absent, the utility should continue as a marked draft, using `No aplica.` for absent inapplicable content and `Pendiente de confirmar:` for missing applicable content.

Restricted production values remain out of ordinary SDD evidence. Final-document-only values may appear only when explicitly supplied by the user for the final manual operational document and must not be backfilled into design, tasks, review, verify, archive, examples, tests, or fixtures.

## Secure Development Design

### Classification and Changed Surface

Classification: security-impacting. The changed surface is not an application runtime; it is the SDD process that controls how agents collect, review, verify, archive, and document operational evidence. The change touches active Markdown instruction contracts under `skills/`, shared safe-evidence guidance, and OpenSpec specs. It does not touch authentication, sessions, database access, production integrations, runtime file upload/download behavior, or application logging code.

The relevant security concern is evidence safety: removing a mandatory readiness contract must not permit leakage of production hostnames, IPs, ports, SID/service names, credentials, tokens, raw logs, sensitive payloads, full ID lists, generated bytes, PAN, PII, or confidential operational context into ordinary SDD artifacts. Omitted security categories remain reviewable by `sdd-review-security`; design must not emit compact matrices, Source ID matrices, all-row `N/A` bookkeeping, or machine-readable applicability fields.

### Secrets and Restricted Operational Data Rules

Implementation must remove references to the deleted readiness contract while keeping the safe-evidence policy in `skills/_shared/sdd-security-contract.md` and affected phase skills. Evidence owners are `test-design` for planned checks, `tasks` for implementation tasks, `apply` for static/search evidence, `review-security` for leakage verdicts, and `verify`/`archive` for preserving non-blocking evidence and warnings. Expected evidence is review-safe: file paths, section names, search summaries, and sanitized excerpts only. Residual risk is over-removal of safe-evidence language; review-security must block unsafe evidence or incomplete security proof.

### Sensitive Logging and Generated Artifact Rules

Operational logs, traces, errors, generated files, exported contents, and diagrams may be mentioned only as safe summaries or placeholders in ordinary SDD evidence. The implementation must not introduce examples containing raw logs, stack traces with sensitive payloads, full generated bytes, full exports, or full ID lists. Expected evidence is static inspection of updated skills/specs and search results showing the restricted-data boundary remains present. Archive should preserve any warnings without copying sensitive content.

### Files and Documentation Boundary Rules

Deleting the shared contract is intentional, but archived artifacts are not rewritten. Active files must not instruct phases to read the deleted path. The manual operational document may include explicitly user-provided final operational values only in the generated final document; those values must never be copied back into SDD artifacts, examples, tests, review reports, verify reports, or archive reports. Evidence owners are `apply`, `review`, `review-security`, and `archive`. Expected evidence is changed-file references and static confirmation that active contracts distinguish ordinary SDD evidence from final-document-only values.

### Exception and Evidence Policy

No exceptions are planned. If implementation cannot preserve safe-evidence protections while deleting the readiness contract, the change must route to `resolve-blockers` instead of approving a partial exception. All evidence must be review-safe and must cite paths, sections, command summaries, or redacted placeholders only.
