# Delta for sdd-test-design-workflow

## ADDED Requirements

### Requirement: Operational Readiness Test Planning

`sdd-test-design` MUST plan static, documentary, manual, or automated checks for operational-readiness evidence from `design.md#operational-readiness`. Checks MUST validate placeholder compliance, evidence traceability, monitoring mechanism coverage, and safe-evidence boundaries. Missing runtime tooling MUST NOT prevent static or documentary checks.

#### Scenario: Readiness checks are planned

- GIVEN design includes operational-readiness strategy
- WHEN `test-design.md` is produced
- THEN it MUST list checks for logs, monitoring mechanisms, administration, reprocessing, ownership, and unresolved gaps
- AND each mandatory check MUST state expected evidence.

#### Scenario: Placeholder compliance is planned

- GIVEN a readiness field may be unresolved or inapplicable
- WHEN test design defines checks
- THEN it MUST require exact `Pendiente de confirmar:` or exact `No aplica.` validation.

#### Scenario: No runtime runner exists

- GIVEN repository testing capabilities report no executable runner
- WHEN readiness validation is planned
- THEN test design MUST use static or documentary evidence where appropriate.

### Requirement: Operational Data Safety Checks

Test design MUST plan checks that ordinary SDD evidence, code, tests, fixtures, and examples do not contain restricted production operational data. These checks MUST distinguish safe SDD evidence from final operational documentation inputs.

#### Scenario: Restricted data check is required

- GIVEN readiness evidence references operational infrastructure
- WHEN test design records validation work
- THEN it MUST include checks for hostnames, IPs, ports, SID/service names, secrets, payloads, full ID lists, and generated bytes.
