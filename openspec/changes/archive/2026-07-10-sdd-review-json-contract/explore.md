# Exploration: sdd-review JSON Contract

## Current State

`sdd-review` currently treats `review-report.md` as the durable review artifact. In OpenSpec mode it writes `openspec/changes/{change-name}/review-report.md`; in Engram/hybrid mode it uses `sdd/{change-name}/review`. The report template defines the Markdown structure directly: verdict table, blocking summary, evidence summary, and a 96-row review matrix whose header must exactly be `Item`, `Artifact/Deliverable`, `Requirement`, `Reviewer`, `Standard`, `Severity`, `Complies`, `Affected Requirement`, `Evidence Location`, `Observations/Comments`.

The 96-control catalog is currently Markdown-only at `skills/sdd-review/references/control-catalog.md`. It preserves stable `REV-CORP-001..REV-CORP-096` IDs, source item mapping, reviewer, standard, severity, default compliance value, evidence hints, and notes. The OpenSpec spec (`openspec/specs/sdd-review-workflow/spec.md`) requires `review-report.md` to include verdict, matrix, evidence summary, exact matrix columns, stable IDs, N/A evidence, and routing semantics.

`sdd-review-security` already demonstrates the desired canonical-data pattern: `security-guideline-catalog.operational.json` is the canonical machine-readable source for automation and audit, while Markdown is a derived human/audit view. Its JSON includes schema identity/version, snapshot/catalog metadata, reporting defaults, allowed vocabulary, row inventories, mappings, counts, and presentation hints. That pattern is a strong fit for a canonical `sdd-review` report contract.

## Affected Areas

- `skills/sdd-review/SKILL.md` — must change the produced artifact contract from Markdown-only to JSON canonical plus derived Markdown, while preserving downstream `review-report.md` behavior.
- `skills/sdd-review/references/report-template.md` — should become the derived Markdown presentation contract and document which JSON fields render each section.
- `skills/sdd-review/references/control-catalog.md` — remains the current catalog source unless a later design also creates a JSON catalog; this change can consume it while emitting canonical report JSON.
- `openspec/specs/sdd-review-workflow/spec.md` — needs requirements for a canonical JSON report artifact and Markdown derivation/compatibility.
- `skills/_shared/persistence-contract.md` — needs resolver/status wording for the additional review JSON artifact in OpenSpec/Engram/hybrid modes without removing the existing `review-report.md` reference consumed by downstream phases.
- `skills/_shared/sdd-post-apply-gates.md` — downstream review evidence consumption should acknowledge that `review-report.md` remains the compatibility presentation, while the JSON is the review source of truth when present.
- `skills/sdd-review-security/SKILL.md` — may need only minimal wording so it continues to consume `review-report.md` and can optionally cite the JSON metadata if provided.

## Approaches

1. **Canonical JSON report plus derived Markdown compatibility** — `sdd-review` writes `review-report.json` as the source of truth and derives `review-report.md` from it in the same phase.
   - Pros: matches user intent; supports future Excel generation; keeps downstream phases working; allows schema/version validation; mirrors the security JSON-canonical pattern.
   - Cons: introduces dual artifacts and requires consistency checks so Markdown cannot drift from JSON.
   - Effort: Medium.

2. **Embed JSON block inside Markdown only** — keep one `review-report.md` file and place canonical JSON in a fenced block.
   - Pros: fewer files; existing downstream paths remain unchanged.
   - Cons: poor as an automation contract; future Excel script would need Markdown extraction; easier to corrupt formatting; less aligned with the security catalog pattern.
   - Effort: Low, but strategically weak.

3. **Replace Markdown with JSON only** — stop producing `review-report.md` and require all downstream phases to consume JSON.
   - Pros: simplest source-of-truth story.
   - Cons: breaks current persistence contract, OpenSpec spec, security review, verify, archive, and human audit expectations that read `review-report.md`.
   - Effort: High and unnecessarily disruptive.

## Recommendation

Use Approach 1. The proposal should define `review-report.json` as the canonical contract/source of truth and keep `review-report.md` as a derived presentation generated by `sdd-review` during the same phase. The Markdown output should remain at the current path and retain the exact existing required sections and matrix header, so `sdd-review-security`, `sdd-verify`, and `sdd-archive` do not break.

The JSON should be designed for future Excel generation but not implement Excel generation in this change. Recommended JSON shape should include:

- schema metadata: `schemaName`, `schemaVersion`, `changeName`, `artifactKind`, `generatedAt`, `sourceCatalogRef`, `derivedMarkdownRef`.
- verdict/routing: `verdict`, `blockingFailureCount`, `nonBlockingFindingCount`, `nextRecommended`.
- evidence summaries: inspected inputs, changed-file context, runtime/unavailable tooling notes, security-boundary handoff, operational evidence summary.
- `blockingSummary[]` rows matching the Markdown blocking table.
- `reviewMatrix[]` with exactly 96 entries and stable fields matching the current Markdown matrix columns.
- vocabulary and validation metadata: allowed complies values, expected row count, catalog validation result, required Markdown presentation sections, and any future presentation hints needed by Excel.

Persistence should extend, not replace, existing references. In OpenSpec mode, the review phase should persist both:

- canonical: `openspec/changes/{change-name}/review-report.json`
- derived compatibility presentation: `openspec/changes/{change-name}/review-report.md`

In Engram/hybrid mode, the existing `sdd/{change-name}/review` key should either contain or reference the canonical JSON and derived Markdown metadata, while OpenSpec hybrid still writes both files. The persistence contract should explicitly state that downstream phases may continue resolving `review-report.md` as the stable presentation artifact; JSON is the authoritative source when available.

## Non-Goals for Proposal

- Do not implement Excel generation or Python scripts in this change.
- Do not remove or rename `review-report.md`.
- Do not change the fixed 96-control matrix columns, row count, or `Complies` vocabulary.
- Do not transfer security-review ownership from `sdd-review-security` to `sdd-review`.
- Do not require downstream phases to parse Excel or any future presentation artifact.

## Risks

- Drift risk: if Markdown is editable independently after generation, it can disagree with canonical JSON. The review phase should validate/read back both and treat JSON as source of truth.
- Contract ambiguity: persistence/status artifacts currently name only `reviewReport`; proposal/design should decide whether to add `reviewReportJson` or keep JSON as metadata under the review report artifact ref.
- Backward compatibility risk: downstream phases currently consume `review-report.md`; this change must keep that file present and non-blocking for existing flows.
- Schema stability risk: future Excel generation needs row/order/presentation hints, but overfitting the JSON to Excel too early would pollute the review contract. Keep Excel-specific rendering out of scope and store only general presentation metadata.

## Ready for Proposal

Yes. The proposal should scope a backward-compatible canonical JSON review report contract, derived Markdown generation inside `sdd-review`, persistence contract updates for the extra JSON artifact, and OpenSpec spec updates proving downstream compatibility.
