## Activation Contract

Run manually after an SDD change has been archived and the user asks for a technical document, documentación técnica, or `sdd-technical-doc`.

This utility is manual, user-invocable, post-archive, and archive-consuming only. It is not an SDD DAG phase and must not change phase order, status routing, verify gates, archive gates, dependencies, task state, or archived evidence.

## Hard Rules

- Use `skills/sdd-technical-doc/assets/technical-document-template.md` as the mandatory output structure.
- Generate exactly one neutral professional Spanish Markdown document, or return it inline when no target path is supplied.
- Use only archived SDD evidence plus explicit user-provided final-document context; never generate from active change artifacts or current code drift.
- Do not invent product, owner, functional, runtime, database, security, integration, object, or reference details.
- For inapplicable sections, write exactly `No aplica.`.
- For applicable but missing information, write exactly `Información no disponible en la evidencia archivada.`.
- Final references may include only FCTI, story, `.pks`, `.pkb`, and `.sql` package/table sources created or modified by the development. Exclude installer scripts.
- Section `1.2 Alcance` must include the required inventory table and only the allowed `Tipo` and `Operacion` enum values from the template.
- Section `2.1 Vistas 4 + 1` must include architecture diagrams when archived evidence supports them: class diagram in `2.1.1 Vista Lógica`, component diagram in `2.2.2 Vista de Desarrollo`, sequence diagram in `2.3.3 Vista de Proceso`, deployment diagram in `2.3.4 Vista Física`, and use-case diagram in `2.3.5 Vista de Escenarios`.
- Diagrams must be Markdown-renderable Mermaid blocks, derived only from archived evidence plus explicit user-provided final-document context; if a required diagram cannot be supported by evidence, write the unavailable-information marker instead of inventing nodes, actors, classes, components, infrastructure, or relationships.
- Summarize evidence by path, section, or sanitized description only. Do not copy secrets, credentials, tokens, private keys, connection strings, PAN, PII, confidential client values, raw payloads, production hosts/IPs/ports/SID values, full ID lists, or generated bytes into SDD artifacts, examples, summaries, or logs.
- User-provided final-document-only values may appear only in the generated final document and must never be backfilled into SDD artifacts, examples, tests, reports, or archived design evidence.

## Decision Gates

| Condition | Action |
| --- | --- |
| Archived change cannot be found | Stop and ask for the archive path or change name. |
| User asks to run before archive | Stop; this utility consumes archived changes only. |
| Hybrid backends differ materially | Stop and ask which backend is authoritative. |
| Evidence proves a section is out of scope | Write `No aplica.`. |
| Evidence is missing for applicable content | Write `Información no disponible en la evidencia archivada.`. |
| Diagram details are not supported by archived evidence or explicit final-document context | Write `Información no disponible en la evidencia archivada.` for that diagram. |
| Inventory value is outside allowed enums | Correct only from archived evidence or use the unavailable-information marker. |
| Target path exists | Read existing content before overwrite decisions. |
| No target path is supplied | Return the document inline and suggest an archive-local Markdown path. |

## Execution Steps

1. Resolve the archived change name or archive folder and selected backend.
2. Read `skills/sdd-technical-doc/assets/technical-document-template.md`.
3. Read archived evidence only: proposal, specs, design, test-design, tasks, review, review-security, verify, archive report, and safe archived source references when present.
4. Extract only review-safe facts needed for the template; keep restricted values out of summaries and SDD evidence.
5. Fill every template section, using the exact inapplicable and unavailable markers where required.
6. Add the required 4+1 view diagrams in their target sections when supported by evidence/context; otherwise use the unavailable-information marker for the unsupported diagram.
7. Validate final references, the `1.2 Alcance` inventory enum values, and diagram placement.
8. If writing to disk, prefer `openspec/changes/archive/YYYY-MM-DD-{change-name}/technical-document.md` unless the user provided another safe Markdown path; read existing content first.
9. Return a concise summary with output location, evidence sources, marker counts, diagram coverage, and confirmation that no SDD state was mutated.

## Output Contract

Return:
- Generated document path, or `inline only`.
- Archived evidence sources read.
- Count/list of `No aplica.` and `Información no disponible en la evidencia archivada.` markers.
- Diagram coverage for `2.1.1`, `2.2.2`, `2.3.3`, `2.3.4`, and `2.3.5`, including any diagram replaced by the unavailable-information marker.
- Confirmation that exactly one Spanish Markdown document was generated.
- Confirmation that SDD DAG/status/verify/archive state and archived evidence were not mutated.
