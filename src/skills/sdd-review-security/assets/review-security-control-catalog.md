## Source-Row Usage Contract

- Active security review validates exactly 155 concrete Source IDs under `sourceRowValidation.rows`.
- Missing, duplicate, or unknown Source IDs block.
- Navigation and grouped non-applicability use source-row fields only: `controlDomain`, `corporateSection`, `repoProfiles`, `runtimeSurface`, or `dataSurface`.
- Grouped non-applicability is presentation only; every row keeps its own `applies`, `complies`, `justification`, `evidenceType`, `evidenceLocation`, `finding`, `ownerPhase`, and `route` in JSON.
- Evidence must cite review-safe paths, section anchors, sanitized summaries, command summaries, or redacted placeholders. Do not include secrets, PII, PAN, tokens, connection strings, private keys, raw logs, generated bytes, or confidential values.

## Expected Evidence Model

- `design-control`: architecture or data-flow decision that satisfies an applicable source row.
- `implementation-reference`: file, function, config, prompt, or artifact section where the control is implemented.
- `test-design-check`: planned automated, static, or manual check in `test-design.md`.
- `verification-evidence`: result recorded by `sdd-verify`.
- `security-review-evidence`: row-level implementation validation recorded by `sdd-review-security` in canonical `review-security-report.json`.
- `approved-exception`: explicit risk acceptance when mandatory evidence cannot be produced.

Mandatory applicable source rows block archive unless evidence is complete or an approved exception follows `skills/_shared/sdd-security-contract.md`.
